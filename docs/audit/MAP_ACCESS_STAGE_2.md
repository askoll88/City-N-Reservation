# Этап 2: система доступа к локациям

Дата: 2026-04-28

## Что сделано

Добавлен единый слой проверки входа в локации: `game/map_access.py`.

Главная цель: вход в зоны больше не должен проверяться разрозненно в разных обработчиках. Любая будущая точка входа — текстовая команда, кнопка карты, callback, эвент, рейд — должна использовать один контракт:

```python
from game.map_access import can_enter_location

access = can_enter_location(player, location_id)
if access.blocked:
    show(access.format_message())
```

## Важное решение по балансу

Уровень и ранг являются жесткими воротами.

Если игрок ниже требуемого уровня или ранга, вход запрещен. Это сделано осознанно: если пустить низкорангового игрока в высокоуровневую зону, он может умереть, но сохранить часть высокорангового лута, и это ломает экономику и progression gate.

## Что проверяет `can_enter_location`

### Базовые требования карты

Берутся из `game.map_schema.get_map_location()`:

- `level_min` — минимальный уровень входа;
- `level_max` — рекомендуемый верхний диапазон, сейчас дает предупреждение, но не блокирует;
- `requires` — явные требования доступа.

### Ранг

Минимальный ранг выводится из `level_min` по таблице рангов игрока.

Пример:

- зона `level_min=1-4` требует ранг 1;
- зона `level_min=5-8` требует ранг 2;
- зона `level_min=9-13` требует ранг 3.

Если в `requires` задан `rank_tier`, берется максимум между вычисленным рангом и явным требованием.

### Поддержанные требования `requires`

Сейчас поддерживаются:

- `level` / `level_min`
- `rank_tier`
- `key` / `keys`
- `item` / `items`
- `flag` / `flags`
- `quest_flag` / `quest_flags`
- `reputation`
- `equipped`
- `radiation_max`
- `artifact_slots`
- `money`
- `defense`
- `total_defense`

## Как трактуются ключи

`key` и `keys` проверяются гибко:

- предмет в инвентаре с таким названием;
- пользовательский флаг с таким названием;
- пользовательский флаг `key:<name>`;
- пользовательский флаг `key_<name>`.

Это оставляет свободу для будущей реализации ключей: как предметы, как флаги, или как системные доступы.

## Как трактуется экипировка

`equipped` проверяет имена надетых предметов:

- оружие;
- броня по слотам;
- рюкзак;
- устройство;
- мешочек гильз;
- артефакты.

Для будущих зон можно писать:

```python
"requires": {
    "equipped": ["Дозиметр", "Бронежилет"],
    "defense": 18
}
```

## UI-поведение

Проверка подключена в `handlers/location.py` внутри `go_to_location`.

Если вход закрыт, игрок остается на текущей локации и получает сообщение:

```text
⛔ Проход закрыт.

Локация: ...

Не хватает:
• уровень 5+ (сейчас 3)
• ранг 2+ (сейчас 1)
• ключ: ...
```

Это лучше, чем просто "нельзя": игрок сразу понимает, что прокачивать или искать.

## Что не сломано

- Старые `LOCATIONS` остались совместимыми.
- Текущие безопасные переходы по городу продолжают работать.
- КПП остается главным входом в Зону.
- Текущие дороги не получили неожиданных новых требований, кроме уже размеченного `level_min`.
- Старые emission-блокировки safe-зон остались на месте.
- Legacy-блокировка убежища для newbie-kit осталась на месте.

## Тесты

Добавлен файл: `tests/test_map_access.py`.

Покрыто:

- текущие маршруты допускают игрока, если уровень/ранг подходят;
- неизвестная локация блокируется;
- `level_min` блокирует вход;
- ранг выводится из уровня зоны;
- низкий ранг блокирует high-tier зону даже при достаточном уровне;
- ключи, предметы, флаги, репутация, экипировка, радиация, деньги и защита проверяются;
- сообщение причин содержит конкретные недостающие условия.
