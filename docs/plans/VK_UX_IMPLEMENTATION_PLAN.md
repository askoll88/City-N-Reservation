# План внедрения UX-возможностей VK API

Дата: 2026-04-28

Цель: использовать возможности VK API шире, чем `messages.send + keyboard`, чтобы игра меньше засоряла чат, быстрее реагировала на действия и лучше показывала важные состояния.

## Ограничение по форматированию

В обычных сообщениях VK-бота не полагаемся на markdown/HTML:

- не используем `**жирный**`;
- не используем `<b>...</b>`;
- не используем markdown-ссылки и code blocks;
- UI строим обычным текстом: переносы строк, короткие блоки, emoji, ASCII/Unicode-разделители.

## Этап 1. Базовая обёртка отправки сообщений

Статус: реализовано в базовом виде.

Что сделать:

- Ввести единый helper для отправки сообщений. ✅ `infra/vk_messages.py`
- По умолчанию прокидывать безопасные параметры: ✅
  - `disable_mentions=1`;
  - `dont_parse_links=1`.
- Централизовать fallback при ошибках VK API. ✅ retry без attachment
- Поддержать единый способ отправки:
  - обычного текста; ✅
  - текста с клавиатурой; ✅
  - текста с attachment; ✅
  - callback/snackbar-ответа. ✅

Что подключено:

- `infra/state_manager.py`: `try_edit_or_send` теперь использует общий transport helper.
- `handlers/location.py`: отправка локаций с картинками использует общий fallback без attachment.
- `main.py`: callback-ответы и возврат из callback используют общий helper.
- `tests/test_vk_messages.py`: проверяет safe defaults, keyboard payload, fallback без attachment и snackbar payload.

Следующая чистка:

- При доработке каждого обработчика заменять прямые `vk.messages.send(...)` на `vk_messages.send(...)`.
- После перевода крупных модулей добавить статическую проверку, запрещающую новые прямые `vk.messages.send` вне transport/helper-слоя.

Зачем:

- меньше дублирования `vk.messages.send`;
- безопаснее пользовательский/игровой текст;
- проще внедрять `messages.edit`, attachments и callback-ответы.

Кандидаты на изменение:

- `infra/state_manager.py`;
- `handlers/location.py`;
- `main.py`;
- массовые `vk.messages.send` в `handlers/*`.

## Этап 2. Callback-кнопки и snackbar-ответы

Статус: запланировано.

Что сделать:

- Перевести часть действий с текстовых кнопок на callback-кнопки.
- Использовать `messages.sendMessageEventAnswer` для коротких ответов без нового сообщения в чат.

Приоритетные экраны:

- карта и переходы;
- рынок: страницы, сортировка, покупка, отмена;
- инвентарь: разделы, осмотр, надеть, снять, выбросить;
- бой: атака, навыки, инвентарь, убежать;
- аномалии: обойти, бросить гильзу, отступить;
- подтверждения: покупка, лечение, риск при выбросе.

Зачем:

- игрок меньше пишет руками;
- чат не засоряется кнопочными командами;
- быстрые ошибки показываются коротким всплывающим ответом.

## Этап 3. Редактирование сообщений

Статус: запланировано.

Что сделать:

- Использовать `messages.edit` для экранов, где пользователь листает или переключает состояние.
- Хранить `conversation_message_id`/`message_id` активного UI-сообщения там, где это нужно.
- При невозможности редактирования откатываться на обычную отправку нового сообщения.

Приоритетные экраны:

- карта регионов;
- список лотов рынка;
- мои лоты/мои сделки;
- инвентарь по разделам;
- просмотр предмета;
- ежедневные задания;
- бой, если текущая модель сообщений позволит обновлять HUD без потери читаемости.

Зачем:

- меньше спама в диалоге;
- экраны ведут себя как интерфейс, а не как поток логов;
- листание рынка/карты становится заметно удобнее.

## Этап 5. Карусели и template

Статус: эксперимент.

Где пробовать:

- витрина магазина;
- P2P-рынок;
- выбор класса;
- выбор NPC;
- карта регионов.

Ограничения:

- проверить стабильность на мобильном и десктопном VK;

Зачем:

- лучше показывать товары и лоты;
- меньше длинных списков в одном сообщении;
- визуально отделить карточки предметов.
